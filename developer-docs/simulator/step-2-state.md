This document covers the **State** element — step 2 of the Structure
group in the wizard (see `abm-wizard-framework.md` for how this fits into

**Audience:** maintainers working on the state/runtime layer, the
simulation loop, or anything that reads/writes agent state during a run.

**Status:** implemented (ticket S-02). Not yet wired into the full

## Scope

**In scope for this step:**

- `AgentState`: runtime representation of a single agent's state
- Distribution sampling via NumPy (one array per attribute, not per agent)
- `initialise_population`: takes an `AgentType` (authoring layer) and
  produces a list of `AgentState` instances (runtime layer)
- Snapshot mechanism: copy state at the start of a step, apply updates to
  the copy, swap at the end of the step to avoid order-dependency bugs
- Mutation tracking: count mutations per agent (infrastructure for a
  future quota mode — the *rules* that trigger mutations live in
  Behaviour)
- Descriptive `KeyError` on undefined attribute access

**Explicitly out of scope for this step:**

- Trigger mode (event-driven state changes based on t-1 conditions)
- Quota mode rules (what triggers the mutation limit) — Behaviour territory
- All-mode eligibility checks — also Behaviour territory
- SoA (Structure of Arrays) conversion — a separate module (`soa.py`)

## Relationship to Step 1 (Agent)

`AgentType` (Step 1) is the **authoring/config layer** — the blueprint a
user builds in the wizard. `AgentState` (this step) is the **runtime
layer** — the live, per-agent instance produced from that blueprint when
a simulation starts. This split keeps the wizard's config model free of
simulation-run concerns (agent IDs, mutation counts, live values), and
keeps the runtime model free of config/validation concerns.

## Data model

### `AgentState`

Runtime state for a single agent instance.

- `agent_id: str` — UUID, auto-generated
- `agent_type_name: str` — links back to the originating `AgentType.name`
- `state: dict[str, Any]` — keyed by attribute name from that agent
  type's `AttributeDefinition` list
- `mutation_count: int` — incremented on every successful `set()` call

**Methods:**

- `get(attr_name) -> Any` — reads an attribute; raises `KeyError` if the
  attribute name isn't in `state`
- `set(attr_name, value) -> None` — writes an attribute; raises
  `KeyError` if the attribute name isn't in `state`. **New keys cannot be
  added at runtime** — only attributes that were part of the original
  `AgentType` schema can be set. Increments `mutation_count` on success.
- `snapshot() -> AgentState` — returns a deep copy of the agent, for use
  in the snapshot update pattern (see below)

Both `get` and `set` raise a `KeyError` with the agent ID, agent type
name, the missing attribute name, and the list of available attributes —
intended to be directly actionable without needing a debugger.

## Distribution sampling

Sampling happens **per attribute, across the whole population, in one
NumPy call** — not per agent. `_sample_distribution(attr, count)` returns
a NumPy array of length `count` for a single attribute.

- The output dtype is inferred from the **attribute's declared type**
  (`AttributeType`), not from the distribution itself — e.g. a `normal`
  distribution on an `INT` attribute is sampled as floats then rounded
  and cast to `int`.
- `fixed` broadcasts a single value to a constant array.
- `uniform` on an `INT` attribute uses `np.random.randint` with `high+1`
  (since `randint` is exclusive of the upper bound).
- `categorical` draws from `np.random.choice` using normalised weights.
- Any distribution type not explicitly handled raises:
  `attribute '<name>': unsupported distribution type '<class name>'`

## Population initialisation

`initialise_population(agent_type: AgentType) -> list[AgentState]`

1. For each attribute on the `AgentType`, sample `count` values as one
   NumPy array (or generate `None` for every agent if no distribution is
   defined yet — see below).
2. Build one `AgentState` per agent by slicing the *i*-th value out of
   each attribute's array.
3. NumPy scalar values are converted to native Python types via `.item()`
   before being stored in `state`.

**Attributes without a distribution** (e.g. a schema-only heterogeneous
attribute still awaiting bulk import — see `PopulationStatus.SCHEMA_ONLY`
/ `PENDING_IMPORT` in Step 1) are initialised to `None` for every agent,
and a warning is printed. This means a population *can* be constructed
from an incomplete `AgentType` — nothing blocks it — but the resulting
state will contain `None` values that downstream Behaviour logic needs to
handle.

## Snapshot update mechanism

The intended simulation-loop usage:

```python
snapshots = [agent.snapshot() for agent in population]
# ... behaviour rules read from `population`, write updates to `snapshots` ...
population = apply_snapshot_updates(population, snapshots)
```

This exists specifically to avoid **order-dependency bugs**: without it,
agent A's update at step *t* could affect agent B's decision at the same
step *t*, depending on which agent happens to be processed first.
Snapshotting first and swapping at the end guarantees every agent's
decision within a step is based on the same pre-step state.

`apply_snapshot_updates(current, updates)`:

- Requires `len(current) == len(updates)`, else raises
  `ValueError: population size mismatch: current=<n>, updates=<m>`
- Overwrites each agent's `state` and `mutation_count` from the
  corresponding update (matched by list position, not by `agent_id` —
  see Troubleshooting)

---

## Mutation tracking

`mutation_count` increments by 1 on every successful `set()` call. This
is **per-agent only** at present — the module docstring mentions
population-wide tracking as part of "quota mode infrastructure," but
there is no population-level aggregation implemented yet. If quota mode
needs a population-wide count, that aggregation currently has to be done
by the caller (e.g. `sum(a.mutation_count for a in population)`).

## How to use

```python
from agents.agents import AgentType
from agents.attributeDefinition import AttributeDefinition
from agents.distributions import NormalDistribution
from agents.models import AttributeType, PopulationMethod, GenerationMode
from state import initialise_population, apply_snapshot_updates

# Step 1: build the blueprint
agent_type = AgentType(
    name="Citizen",
    count=100,
    generation_mode=GenerationMode.HETEROGENEOUS,
    attributes=[
        AttributeDefinition(
            name="age",
            type=AttributeType.INT,
            population_method=PopulationMethod.DISTRIBUTION,
            distribution=NormalDistribution(mean=40, stddev=12),
        ),
    ],
)

# Step 2: initialise runtime state from the blueprint
population = initialise_population(agent_type)

# Reading/writing an individual agent's state
agent = population[0]
print(agent.get("age"))
agent.set("age", agent.get("age") + 1)

# One simulation step using the snapshot pattern
snapshots = [a.snapshot() for a in population]
for snap in snapshots:
    snap.set("age", snap.get("age") + 1)  # behaviour logic would live here
population = apply_snapshot_updates(population, snapshots)
```

## Troubleshooting

**`KeyError` when writing to what looks like a valid attribute**
`set()` only allows updating attributes that already exist in `state` —
i.e. attributes that were part of the originating `AgentType`'s
`attributes` list at initialisation time. New keys cannot be introduced
at runtime. If Behaviour logic needs to add a wholly new piece of state
mid-simulation, that's not supported by this model as-is; the attribute
needs to exist (even as `None`) from initialisation.

**Attribute values are `None` partway through a run**
This happens when `initialise_population` was called against an
`AgentType` where an attribute had no distribution defined (status
`SCHEMA_ONLY` or `PENDING_IMPORT` in Step 1). Nothing blocks population
initialisation from an incomplete blueprint — only a printed warning is
emitted, not an exception. If a downstream calculation fails on `None`,
check the originating `AgentType`'s attribute statuses before assuming
it's a runtime bug.

**`ValueError: population size mismatch` in `apply_snapshot_updates`**
`current` and `updates` must be the same length. This usually means
agents were added or removed *after* snapshotting but before the update
was applied — e.g. an Agent lifecycle rule (reproduction/removal) ran
mid-step outside the snapshot/apply boundary. Lifecycle changes should be
applied outside this pattern, not interleaved with it.

**Snapshot updates silently apply to the wrong agent**
`apply_snapshot_updates` matches `current` and `updates` **by list
position**, not by `agent_id`. If any code re-orders, filters, or
re-sorts either list between `snapshot()` and `apply_snapshot_updates()`,
updates will be applied to the wrong agents with no error raised. Treat
the snapshot list as strictly positionally-paired with the original.

**Unsupported distribution type error during initialisation**
`_sample_distribution` raises `ValueError` for any distribution class it
doesn't explicitly branch on. If new distribution types are added to the
Step 1 model without a matching branch here, population initialisation
will fail for any attribute using them. See the open item above re:
`BinomialDistribution` / `PoissonDistribution` — confirm these are
handled consistently between the two modules.

**Categorical attribute values behave unexpectedly after `.item()`
conversion**
Numeric NumPy scalars convert cleanly to native Python types via
`.item()`. String/categorical draws from `np.random.choice` may not
follow exactly the same path depending on NumPy version — worth a
spot-check that categorical attribute values end up as plain `str` in
`state`, not a NumPy string type, if anything downstream does strict
type checks.
